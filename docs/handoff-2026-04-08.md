## 2. Sobre el proyecto

### 2.1 Qué es

Una aplicación web mobile-first en PHP 8.2 + SQLite para gestionar la limpieza de habitaciones de los dos hoteles de Atankalama Corp. Es un **reemplazo de Flexkeeping**, un software comercial que les cuesta USD $450/mes (≈ CLP $3.240.000/año).

### 2.2 Por qué

Flexkeeping cuesta caro y no está adaptado al flujo del hotel. El equipo TI interno puede absorber el desarrollo (costo marginal cero). El proyecto se va a presentar a la gerencia como un MVP funcional.

### 2.3 Quién lo va a codificar

**Claude Code** (la versión agente de Claude, no esta conversación) corriendo en VS Code en la máquina Windows local de Nicolás. La conversación actual es de **planificación y diseño**, no de codificación. La codificación viene después, en otra fase, una vez tengamos toda la documentación de detalle lista.

### 2.4 Stack confirmado

- **Backend:** PHP 8.2 + SQLite + PDO (sin ORMs externos)
- **Frontend:** PHP nativo como motor de plantillas (sin Blade ni Twig) + Tailwind CSS por CDN + Alpine.js por CDN + Lucide icons + Inter de Google Fonts
- **IA:** Claude API para el copilot conversacional + Web Speech API del navegador para transcripción de voz
- **Hosting:** Hostinger VPS con EasyPanel
- **Integración externa:** Cloudbeds API (PMS hotelero) — dos API keys, una por propiedad
- **Versionado:** Git + GitHub repo público `NicoCalama/atankalama-limpieza`
- **Editor:** VS Code en Windows + extensión Claude Code v2.1.92

### 2.5 Diferenciadores vs Flexkeeping

- **Copilot IA con voz** integrado, respeta permisos por rol
- **RBAC dinámico** — permisos editables sin tocar código
- **Alertas predictivas** — avisa a la supervisora si un trabajador no va a alcanzar a terminar su turno
- **Auditoría con 3 estados** — aprobado, aprobado con observación (trazabilidad granular para coaching), rechazado
- **100% en español chileno** adaptado al flujo y léxico del hotel
- **UX deliberadamente sin generar ansiedad** al personal de limpieza

## 3. Decisiones arquitectónicas tomadas (no negociables ya)

Estas son decisiones que ya se conversaron y cerraron. No hay que volver a discutirlas a menos que Nicolás explícitamente las reabra.

### 3.1 Stack y enfoque
- **Mobile-first real**, no responsive de escritorio escalado hacia abajo
- **Bottom tab bar** en móvil, sidebar en desktop
- **PHP server-rendered con plantillas PHP nativas** (decisión del jefe de Nicolás, opción "más simple para sacar el MVP rápido")
- Tailwind y Alpine **por CDN** — sin build step, sin npm para el frontend
- Idioma **100% español chileno**

### 3.2 Roles y permisos — RBAC dinámico
- 4 roles por defecto pre-cargados pero editables desde la UI: **Admin, Supervisora, Trabajador de limpieza, Recepción**
- **NO se chequean roles directamente en código.** Siempre se chequean **permisos específicos** vía `$usuario->tienePermiso('codigo.permiso')`
- Existe una **pantalla en Ajustes del Admin** con una matriz de checkboxes (filas = permisos, columnas = roles) donde se gestiona todo sin tocar código
- El Admin puede crear roles nuevos (ej: "Jefe de Turno", "Lavandería") sin tocar código
- Los **permisos nuevos** sí requieren código, pero se gestionan con un seeder único `database/seeds/permisos.php`. Al desplegar, aparecen automáticamente en la matriz de Ajustes
- Existe un catálogo inicial de permisos completo en `plan.md` sección 5.3

### 3.3 Auditoría con 3 estados (no 2)

- **Aprobado** — todo bien, habitación queda Clean en Cloudbeds
- **Aprobado con observación** — auditor encontró algo menor, lo resolvió en el momento, deja constancia
  - Flujo UI: al tocar este botón, la pantalla expande el checklist completo del trabajador y el auditor desmarca los ítems específicos donde encontró el problema
  - Trazabilidad granular: queda registrado QUÉ ítem del checklist falló, no solo "hubo observación"
  - El trabajador NO ve esto como rechazo en su historial — ve la habitación como aprobada normal
  - Internamente afecta KPIs a nivel de ítem específico (no de habitación entera)
  - Permite reportes tipo "María Inés tiende a fallar en limpiar espejos del baño" para coaching dirigido
- **Rechazado** — necesita re-limpieza
  - Vuelve a Dirty en Cloudbeds
  - Supervisora recibe alerta y decide a quién reasignar
  - Aparece como "Rechazada" en el historial del trabajador

### 3.4 Persistencia del checklist tap a tap

- Cada vez que el trabajador marca o desmarca un ítem del checklist, se dispara INMEDIATAMENTE un PUT/POST al backend
- Si no hay internet, queda en cola local y sincroniza al volver
- Si la app se cierra (batería, crash, llamada), el progreso NO se pierde
- Al reabrir, la habitación aparece como "Continuar" con todos los checks previamente marcados
- El botón "Habitación terminada" se desbloquea SOLO cuando todos los ítems están marcados

### 3.5 Tracking de tiempo OCULTO al trabajador

- Cada habitación registra `timestamp_inicio` (cuando tocó "Comenzar limpieza") y `timestamp_fin` (cuando tocó "Habitación terminada")
- El trabajador **NUNCA ve estos tiempos** en su pantalla. Ni en home, ni en checklist, ni en historial
- Esta data es exclusivamente para:
  - Cálculo del tiempo promedio personal del trabajador (input del algoritmo de alertas predictivas)
  - Reportes, KPIs, analítica (versión completa en Fase 2)
- Es deliberado para **no generar ansiedad** al personal

### 3.6 Sistema de alertas predictivas (versión MVP)

- El sistema calcula en tiempo real si cada trabajador va a alcanzar a terminar su turno
- Cálculo: `tiempo_estimado_restante = habitaciones_pendientes × tiempo_promedio_personal` vs `tiempo_de_turno_restante`
- Margen de seguridad default: **15 minutos** (configurable desde Ajustes)
- Para trabajadores nuevos sin histórico: usa promedio global del equipo hasta que tenga al menos N limpiezas (default 5)
- Solo visible para la **supervisora** (badge en campana + banner en su Home + bandeja de alertas)
- El **trabajador NUNCA** ve la alerta ni sabe que existe — es deliberado para no generar presión
- Recálculo automático al completar habitaciones y cada 15 min en background
- Guiños a Fase 2 visibles en Ajustes como opciones grises: ponderación por tipo de habitación, push notifications, sugerencias automáticas de reasignación

### 3.7 Filosofía UX — sin generar ansiedad

Decisión muy importante de diseño que se aplica transversalmente: las pantallas del trabajador deliberadamente **ocultan métricas y contadores numéricos** ("te quedan 3", "vas al 67%") porque pueden generar ansiedad. En cambio, mostramos **barras de progreso visuales** que transmiten sensación de avance sin presión.

La presión se gestiona **desde el rol de la supervisora** vía las alertas predictivas (que el trabajador no ve).

### 3.8 Copilot IA — niveles y permisos

- **Nivel 1 (consultas)** y **Nivel 2 (acciones simples)** entran al MVP
- **Nivel 3 (acciones complejas)** queda fuera, reconocido pero respondido con "esa acción aún no está disponible"
- El copilot respeta los **permisos dinámicos** del usuario (no chequea rol, chequea permisos)
- Confirmación visual antes de ejecutar acciones sensibles
- Pide clarificación ante ambigüedad
- Historial persistente por usuario en SQLite, visible en Ajustes
- Acciones del copilot quedan en logs con flag `via_copilot_ia = true`
- Voz vía Web Speech API del navegador (gratis, nativo, suficiente para MVP; migrable a Whisper si hace falta)
- FAB flotante siempre visible en esquina inferior derecha de todas las pantallas
- Abre panel deslizable desde abajo en móvil, panel lateral en desktop

### 3.9 Autenticación

- Login con **RUT chileno** (formato flexible, validación de dígito verificador en cliente)
- **NO hay** checkbox "mantener sesión iniciada" — sesiones largas porque dispositivos son personales
- **No hay** recuperación de contraseña por email — el admin resetea desde su panel y genera contraseña temporal
- Contraseñas temporales: **6 caracteres alfanuméricos sin ambigüedades** (sin 0/O, 1/l/I) — ejemplo `K7M4XP` — fáciles de dictar por WhatsApp
- **Primer login obligatorio** con cambio de contraseña antes de acceder a cualquier otra pantalla
- Toggle día/noche accesible desde la pantalla de login (antes de loguearse)

### 3.10 Turnos

- Cada usuario tiene un **turno asignado** en el momento de creación por el Admin
- Turnos pre-cargados por defecto: **Diurno (08:00-17:00)** y **Nocturno (20:00-05:00)**
- El Admin puede crear/editar turnos desde Ajustes → Turnos
- Casos complejos (cambios puntuales, doblar turno, turnos rotativos) son **Fase 2**

### 3.11 Sincronización con Cloudbeds

- **Lectura**: 2 veces al día (horarios configurables), alineados con turnos mineros (gran parte de los huéspedes son trabajadores de minería)
- **Escritura**: inmediata al marcar habitación como completada (PUT a Cloudbeds)
- **Escritura**: inmediata al rechazar auditoría (vuelve a Dirty)
- Cola de reintentos `cloudbeds_sync_queue` para casos de fallo
- 3 reintentos con backoff exponencial (1s, 2s, 4s)
- Timeout 10s por request
- Todas las llamadas se loggean (sin headers Authorization)
- Dos API keys distintas (una por propiedad), nunca mezclar

### 3.12 Modo de trabajo de Claude Code: Autonomía Híbrida (c3)

- **Autonomía total** para módulos backend mecánicos: schema, modelos, CRUD, RBAC, integración Cloudbeds, alertas predictivas, tests
- **Supervisión por módulo** para UI: Login, Home (4 versiones), checklist, copilot IA, ajustes (matriz RBAC), auditoría, alertas predictivas
- Cuando Claude Code tome decisiones autónomas, debe marcarlas con comentarios `// DECISIÓN AUTÓNOMA: ...`
